# 同步生命周期

## 核心模型

`dotai` 只管理两个方向的数据流，所有操作都围绕**单一真相源**展开：

```
.dotai/rules/*.md（唯一真相源）
    ↓ render（向下分发）
原生格式文件（CLAUDE.md、AGENTS.md 等）
    ↓ import（向上收归）
.dotai/rules/*.md（更新后的真相源）
```

层级之间的规则合并（如用户级与项目级之间的优先级）由 AI IDE 原生机制处理，`dotai` 不介入运行时。

## 资产状态机

### 真相源资产状态

| 状态 | 含义 |
|------|------|
| `clean` | 与最后一次渲染时的快照一致，未发生变动 |
| `modified_local` | 在本地被修改，尚未重新渲染分发 |
| `ahead_of_cloud` | 本地版本比云端版本更新（后续云同步版本适用） |
| `behind_cloud` | 本地版本落后于云端（后续云同步版本适用） |
| `conflicted` | 无法自动解决的双边冲突，需人工介入 |

### 受管目标文件状态

| 状态 | 含义 |
|------|------|
| `rendered_clean` | 当前内容与最后渲染指纹一致，无漂移 |
| `drifted` | 文件被外部直接修改，当前内容与渲染指纹不符 |
| `import_pending` | 漂移已检测到，等待用户发起 `import` 操作 |
| `conflict_review` | 发起 `import` 时检测到双边冲突，阻断写入，等待人工审阅 |

## 核心操作流程

### A. 渲染分发（Render）

```
真相源 → 原生文件
```

步骤：

1. 读取 `.dotai/rules/` 下的规则文件
2. 按目标工具格式调用对应渲染器（Codex / Claude / Skills）
3. 将输出写入对应的原生目标路径
4. 计算并保存本次输出的 SHA-256 指纹至 `.dotai/state/managed-targets.json`
5. 记录资产 ID 到目标文件的映射关系（用于后续反向导入的段落定位）

### B. 漂移检测（Drift Detection）

```
原生文件 → 检查实时指纹 vs 渲染快照指纹
```

步骤：

1. 对受管目标文件计算当前内容的 SHA-256 指纹
2. 与 `.dotai/state/` 中记录的 `lastRenderFingerprint` 比对
3. 不一致则将该文件标记为 `drifted`，并通知用户

### C. 反向导入（Import）

```
原生文件 → 真相源
```

步骤：

1. 用户发起 `dotai import --from <文件路径>`
2. 检查真相源（`.dotai/rules/`）自上次渲染后是否也被修改（检查资产指纹）
3. **若仅原生文件变化（无双边冲突）**：
   - 解析原生文件与渲染快照的差异
   - 尝试将差异段落映射至对应的源资产文件
   - 生成候选新版本，提示用户确认后写入
4. **若真相源也已变化（双边冲突）**：
   - 立即阻断写入，将资产标记为 `conflicted`
   - 打开审阅界面，展示三方差异（原生文件新版、真相源新版、上次渲染快照）
   - 提供"AI 辅助融合"入口，可调用内置 Skill 输出融合草案
   - 用户确认后方可写入

### D. AI 辅助融合（AI-Assisted Merge，可选）

```
冲突的两个版本 → Skill 语义分析 → 融合草案
```

步骤：

1. 将真相源新版本与原生文件新版本作为输入传递给内置 Skill
2. Skill 对两个版本进行语义级分析（非文本行级对比）
3. 输出融合草案，标注各段落来源
4. 用户在审阅界面逐段确认或修改
5. 确认后写入真相源，触发重新渲染分发

## 冲突场景

### 场景 1：仅原生文件被直接修改

```
render → 用户修改 CLAUDE.md → import
```

处理：无双边冲突，系统生成候选版本，提示确认即可。

### 场景 2：原生文件和真相源同时被修改

```
render → 用户 A 修改 CLAUDE.md，用户 B 修改 .dotai/rules/ → import
```

处理：检测到双边冲突，阻断写入，进入三方审阅，可选 AI 辅助融合。

### 场景 3：两个项目仓库同时修改各自的原生文件

```
仓库 A render → 修改 CLAUDE.md → import
仓库 B render → 修改 CLAUDE.md → import（此时与仓库 A 的导入结果冲突）
```

处理：各仓库独立执行 `import` 流程。若涉及同一用户级真相源（`~/.dotai/`），则在第二次 `import` 时检测到双边冲突，进入审阅流程。

### 场景 4：未受管的原生文件被修改

处理：允许将其作为全新资产导入，或由用户手动关联至现有资产后再执行合并。

## 合并置信度

在执行 `import` 时，系统评估合并操作的置信度：

| 等级 | 含义 | 推荐行为 |
|------|------|----------|
| `high` | 仅单一资产变动，影响范围明确 | 可安全执行自动导入 |
| `medium` | 多个资产可能受影响，需要区分段落 | 建议人工审阅后确认 |
| `low` | 改动分布模糊，无法可靠映射至源资产 | 强制人工拆分，或调用 AI 辅助融合 |

## MVP 阶段目标

在第一个支持双向同步的版本中，应确保：

- 支持 `CLAUDE.md` 的反向导入
- 支持 `AGENTS.md` 的反向导入
- 强指纹快照拦截机制完整可用
- 置信度低时强制进入人工审阅，禁止静默融合
- AI 辅助融合作为可选入口可调用，但不强制执行
