# 产品需求文档（PRD）

## 产品形态

`dotai` 是一个面向 AI 编码工具的**双向扩展资产管理器**，负责在各类 AI IDE 的原生配置格式之上提供统一的生命周期管理能力。

它不负责在运行时合并多层级规则——这部分能力已由各 AI IDE 原生机制（如 Codex 的 `AGENTS.md` 作用域继承、Claude 的递归 `CLAUDE.md` 查找）可靠承接。`dotai` 专注于：

- 维护唯一一份可信的"单一真相源"（Single Source of Truth）
- 将这份资产按目标工具格式分发至对应的原生路径
- 安全地将外部对原生文件的修改反向导入并归档至管理库

## 问题

开发者在不断切换：

- 多台设备
- 多个代码仓库
- 多种 AI IDE 与编码代理

每种工具各自维护独立的指令体系：

- `AGENTS.md`（Codex）
- `CLAUDE.md`（Claude）
- `.claude/skills/`（Claude Skills）
- Copilot 指令文件
- Cursor、Trae 等各自的规则文件

这带来的结果是：指令漂移、维护重复，以及缺少一套将项目级经验沉淀为个人长久知识库的清晰路径。

## 愿景

`dotai` 成为 AI 代理指令的**个人操作层**：

- 唯一一份受版本管理的标准真相源
- 按工具格式渲染出多种原生输出
- 明确的层级隔离，各层互不串联
- 跨项目提升与复用
- 本地和云端的版本历史

## 目标用户

- 拥有强烈个人工作流偏好的独立开发者
- 需要跨多仓维护团队开发规范的业务架构师
- 在 Codex、Claude 和各类 IDE 代理之间频繁切换的 AI 重度用户

## 核心设计原则

### 1. 层级物理隔离，原生委派合并

`dotai` 按照与 AI IDE 一致的层级设计维护镜像，层级间严格物理隔离：

| 层级 | 路径 | 职责 |
|------|------|------|
| 用户级 | `~/.dotai/` | 管理个人全局扩展，生成 `~/.claude/CLAUDE.md` 等用户级原生文件 |
| 项目级 | `./.dotai/` | 管理当前项目扩展，生成 `./CLAUDE.md`、`./AGENTS.md` 等项目级原生文件 |
| 子路径级 | `./<dir>/.dotai/`（可选） | 管理子目录覆盖规则 |

层级之间**不由 `dotai` 合并**。运行时的规则继承与覆盖完全交由各 AI IDE 的原生机制处理，`dotai` 不干预这一过程。

### 2. 单一真相源（Single Source of Truth）

`.dotai/` 目录下的资产文件（如 `.dotai/rules/my-style.md`）是唯一的规则底稿。

所有原生输出文件（如 `CLAUDE.md`、`AGENTS.md`）均视为**派生分发产物**，不是权威源头。

链路为：

```
.dotai/rules/*.md（唯一真相源）
    ↓ dotai render
CLAUDE.md（Claude 分发产物）
AGENTS.md（Codex 分发产物）
.claude/skills/（Claude Skills 分发产物）
```

### 3. 三位一体产品形态

`dotai` 以三种形态为用户提供入口，共享同一套底层核心引擎：

- **CLI**：支持在终端、Git Hooks 和自动化脚手架中无头调用
- **VS Code 插件**：提供可视化差异对比、漂移警告和操作面板
- **内置 Skill**：允许在 AI IDE 中直接调用 `dotai` 管理能力（如"请将本次约定的规范沉淀进扩展库"），实现 AI 驱动的扩展闭环管理

### 4. 刚柔并济的双向冲突管控

当原生文件（如 `CLAUDE.md`）被直接修改后，`dotai` 提供双层防护机制将改动安全收归至唯一真相源：

**第一层：强指纹快照拦截（必选）**

每次 `render` 输出时记录内容指纹 Hash。发起反向 `import` 时，系统比对该 Hash：

- 若真相源本身在这期间也被修改，判定为"双边冲突"
- 立即阻断自动写入，强制进入人工审阅流程

**第二层：AI Skill 语义融合（可选，需命令显式调用）**

在人工审阅界面旁提供"AI 辅助合并"入口，调用内置 Skill 对冲突的两份内容进行语义级分析，自动提出融合草案供用户确认。

## 核心流程

### 新仓库启动

1. 执行 `dotai init --project`
2. 将现有的 `AGENTS.md` 或 `CLAUDE.md` 导入为初始资产
3. 在 `.dotai/rules/` 中编写或补充规则
4. 执行 `dotai render --target all`，生成对应原生文件

### 直接修改原生文件后回流

1. 用户直接编辑 `./CLAUDE.md`
2. `dotai` 检测到指纹漂移，将该文件标记为 `drifted`
3. 执行 `dotai import --from ./CLAUDE.md`
4. 若无冲突，自动生成候选版本等待确认
5. 若检测到双边冲突（真相源也被修改），阻断并进入审阅流程，可选择调用 AI Skill 辅助融合
6. 确认后，执行 `dotai render --target all` 重新分发至所有原生文件

### 跨项目提升（项目 → 用户）

1. 执行 `dotai promote --file .dotai/rules/my-rule.md --to user`
2. 规则被复制至 `~/.dotai/rules/`，附带出处元数据
3. 执行用户级渲染，全局生效

## MVP 范围

### 在范围之内

- 项目级镜像初始化与规则管理
- 读取并同步用户级镜像 `~/.dotai/`
- 面向 Codex（`AGENTS.md`）和 Claude（`CLAUDE.md`、`CLAUDE.md` + Skills）的渲染器
- 漂移检测与指纹快照拦截
- 反向导入与冲突审阅流程
- AI Skill 辅助语义合并入口
- VS Code 插件可视化面板
- CLI 无头调用模式

### 不在范围之内

- 云账户同步（后续版本）
- 公共规则包市场
- 完全无损的跨格式自动转换

## 成功衡量标准

- 一份 `.dotai` 资产能正确渲染为 Codex 和 Claude 均可读的原生文件
- 直接修改 `CLAUDE.md` 后，`import` 流程可以安全回收变更且不丢失数据
- 双边冲突时系统能正确阻断并给出可操作的解决路径
- 规则提升操作（项目 → 用户）在 10 秒内完成
- 新设备完成 init 和初次渲染的全流程在 2 分钟内完成
