## 分析目的

对比 `dotai` 当前方向与现有同名产品（`bajinzhi/dotai`），明确差异化路径，避免在错误的维度上竞争。

## `bajinzhi/dotai` 产品概况

根据公开的 Marketplace 描述，该产品定位为：

- 一个 Git 仓库统管所有 AI 工具的配置文件
- 将配置同步至各工具的原生目录
- 支持用户级和项目级部署
- 覆盖 Claude、Copilot、Codex、Trae 等多个工具
- 提供 CLI 和 VS Code 插件两种工作方式
- 功能包括：状态检查、差异对比、配置文件选择、同步控制

冲突处理模式为 `overwrite`、`skip`、`ask`。

## 对方做得好的地方

- **价值主张清晰**：一个仓库，多套工具，原生格式输出。易于理解，易于快速试用。
- **原生格式兼容优先**：不强制私有运行时，契合用户习惯。
- **工具覆盖广度**：宽泛的工具支持列表在市场层面具有显著优势。
- **同步导向的工作流**：状态、差异、配置文件选择等功能已构成可用的第一入口。

## 对方可能存在的差距（推断）

### 1. 以"分发"为主，缺少"知识演进"

公开信息中清楚表述了同步与原生部署能力，未提及：反向导入（import）、规则提升（promote）、语义级资产生命周期管理。

推断：该产品在分发层面较强，在双向版本管控和资产级知识演进上较弱。

### 2. 冲突处理以文件为单位，而非资产为单位

`overwrite / skip / ask` 的模式操作对象是文件，而非逻辑规则块。

推断：若系统以仓库到目标的文件同步为核心，则难以支持：

- 将 `CLAUDE.md` 中一段修改追溯回某个可复用逻辑资产
- 将项目规则提升进个人库后再分发至其他工具格式
- 在双边冲突时以语义而非字节为单位进行分析与融合

### 3. 无规范的双边修改保护机制

若无指纹快照记录，则难以区分"这次改动是来自原生文件的直接编辑"还是"来自上一次同步后的新增内容"。这使得 AI Skill 辅助融合等高级场景几乎无法可靠实现。

## 我们的方向

`bajinzhi/dotai` 是一个多工具配置同步产品。

我们正在构建的是 **AI 扩展资产的双向生命周期管理系统**，核心差异在于：

| 维度 | `bajinzhi/dotai` | 我们的方向 |
|------|------------------|------------|
| 核心定位 | 一个仓库同步配置至多套工具 | 单一真相源驱动，支持双向版本管控与资产级分发 |
| 核心单元 | 文件与目标目录 | 逻辑资产（Asset）+ 受管目标（Managed Target）|
| 冲突处理 | 文件级 `overwrite/skip/ask` | 指纹快照拦截 + AI Skill 辅助语义融合 |
| 层级隔离 | 支持用户级与项目级部署 | 物理隔离，合并委派 AI IDE 原生机制 |
| 反向导入 | 公开信息中未提及 | 核心产品能力 |
| 项目到个人规则提升 | 公开信息中未提及 | 核心产品能力 |
| 跨格式语义传播 | 不明确 | 明确目标 |
| 内置 AI Skill 集成 | 不明确 | 核心产品形态之一 |
| 目标用户 | 有统一配置仓库需求的团队 | 有成长型个人工作流需求的极客与主程 |
| 防御性护城河 | 工具覆盖广度 | 双向生命周期深度 + AI Skill 闭环 |

## 竞争策略

### 不要在错误的维度竞争

若我们仅强调"也支持多工具","也可以同步原生文件"，则是在对方最强的方向上正面硬碰。

### 占据生命周期深度

我们要拥有更难解决的问题：

- 随手改，安全回流
- 版本可追溯，来源透明
- 跨层提升，一次确认，多端生效
- AI 驱动的无缝扩展管理闭环

### 保留原生输出

差异化不意味着放弃原生格式。我们的定位是：

> 原生格式管理 + 真正的双向版本控制

而不是"又一个私有格式工具"。

## 产品定位建议

### 定位句

`dotai` 是一款面向 AI 编码工具的双向扩展资产管理器。它让你以原生格式编辑，安全地反向导入变更，并在多个项目和工具之间进行版本化分发。

### 可用标语

- 以原生格式编辑。集中版本管控。随处分发。
- `CLAUDE.md` 和 `AGENTS.md`，加上真正的双向版本控制。
- 把 AI 配置文件变成可管理、可复用的扩展资产。

### 应避免的表述

- 一个仓库管理所有 AI 工具（与对方直接重叠）
- 零格式转换（同上）
- 配置同步工具（定位过低）

### 应强调的表述

- 双向版本管控
- 单一真相源，原生格式输出
- 个人扩展资产库
- 项目规则提升至用户库
- AI Skill 集成，支持对话式管理

## 开发阶段建议

### 阶段一：验证本地双向闭环

- 项目级 `.dotai` 镜像
- 渲染 `CLAUDE.md` 和 `AGENTS.md`
- 指纹快照漂移检测
- 反向导入与冲突审阅

### 阶段二：接入个人扩展库

- 用户级 `~/.dotai/` 镜像
- 规则提升（项目 → 用户）
- 跨仓选择性同步

### 阶段三：AI Skill 集成

- 内置 Skill 模块
- 对话式规则沉淀与管理
- AI 辅助语义冲突融合

### 阶段四：云端持续性（远期）

- 账户登录
- 云端个人扩展库
- 版本历史与跨设备拉取
